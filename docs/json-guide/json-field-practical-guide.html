😤 日常痛点

痛点 1：新增字段流程漫长

产品经理："我们得给手机加个'快充功率'字段"

开发："好的，我提个 DDL 变更申请"

DBA："等下周变更窗口，先排队"

结果：3 天后才能上线

痛点 2：查询性能差

明明只查 3 个字段，却要扫描 203 列的元数据

执行计划里全是 NULL 值过滤

结果：查询比预期慢 5 倍

痛点 3：代码难以维护

每次 INSERT 要写 200 多个字段名

不同类目的字段还要判断是否为 NULL

结果：代码里全是 if-else

"我们花了一半的时间在改表结构，另一半时间在处理 NULL 值判断"

💡 第二章：JSON 方案，降维打击

🎯 核心思路： 核心字段结构化 + 扩展字段 JSON 化
-- 改造后的表结构：清爽！ CREATE TABLE products_new ( product_id BIGINT, -- 主键 category_id INT, -- 类目 ID（用于筛选） product_name STRING, -- 商品名称 base_price DECIMAL(18,2), -- 基础价格 brand STRING, -- 品牌 attributes JSON, -- ✨ 所有扩展属性！ created_at TIMESTAMP, -- 创建时间 updated_at TIMESTAMP, -- 更新时间 version INT -- 版本号 ); -- 总共：9 列（原来是 203 列！）

📦 数据怎么存？

-- 服装：attributes 里存材质、颜色、尺码 INSERT INTO products_new VALUES ( 1001, 10, '纯棉 T 恤', 99.00, '优衣库', PARSE_JSON('{ "material": "100% 纯棉", "colors": ["白色", "黑色", "灰色"], "sizes": ["S", "M", "L", "XL"], "style": "休闲", "season": ["春", "夏", "秋"], "care_instructions": "机洗，低温烘干" }'), current_timestamp(), current_timestamp(), 1 ); -- 手机：attributes 里存 CPU、内存、屏幕 INSERT INTO products_new VALUES ( 2001, 20, '智能手机 X1', 4999.00, '小米', PARSE_JSON('{ "cpu": "骁龙 8 Gen 2", "ram_gb": 12, "storage_gb": 256, "screen_size_inch": 6.7, "resolution": "2400x1080", "battery_mah": 4500, "camera_mp": 50, "os": "MIUI 14", "5g_support": true }'), current_timestamp(), current_timestamp(), 1 ); -- 图书：attributes 里存作者、ISBN、页数 INSERT INTO products_new VALUES ( 3001, 30, '三体', 38.00, '重庆出版社', PARSE_JSON('{ "author": "刘慈欣", "isbn": "978-7-5366-9293-0", "pages": 496, "publisher": "重庆出版社", "series": "三体系列", "language": "中文" }'), current_timestamp(), current_timestamp(), 1 );
💡 关键变化
  • 所有类目的扩展属性都存进 attributes 字段
  • 不同类目可以有不同的属性结构
  • 新增属性直接写 JSON，无需改表结构
  • 没有 NULL 值浪费，存储更高效

🧪 第三章：真实测试，所有 SQL 已验证

测试环境：云器 Lakehouse 阿里云上海区域 · 实例 f8866243 · 2026-04-10

测试数据：商品表 5 条记录 · 用户事件表 5 条记录

测试 1：查询白色服装 ✅

-- 业务场景：运营要搞"白色系"促销活动 SELECT product_id, product_name, base_price, attributes['colors'] as colors FROM demo_products WHERE category_id = 10 AND CAST(attributes['colors'] AS STRING) LIKE '%白色%' -- 结果：1 条 -- - 纯棉 T 恤 (¥99.00)
⚠️ 注意

JSON 字段访问后需要 CAST(... AS STRING) 才能进行字符串比较

测试 2：查询大内存手机 ✅

-- 业务场景：技术爱好者筛选"12GB 以上内存"的手机 SELECT product_id, product_name, CAST(attributes['ram_gb'] AS INT) as ram_gb FROM demo_products WHERE category_id = 20 AND CAST(attributes['ram_gb'] AS INT) > 8 -- 结果：1 条 -- - 智能手机 X1 (12GB RAM)

测试 3：统计各材质服装数量 ✅

-- 业务场景：采购要看"纯棉"和"化纤"的占比 SELECT CAST(attributes['material'] AS STRING) as material, COUNT(*) as cnt FROM demo_products WHERE category_id = 10 GROUP BY CAST(attributes['material'] AS STRING) -- 结果：2 种材质 -- - 98% 棉 2% 弹性纤维：1 件 -- - 100% 纯棉：1 件
⚠️ 踩坑警告：JSON 字段不能直接用于 GROUP BY，必须先 CAST 转换！

测试 4：查询购买事件的收货地址 ✅

-- 业务场景：客服要查"某订单发到哪里了" SELECT event_id, user_id, event_data['product_name'] as product, event_data['shipping_address']['city'] as city, event_data['shipping_address']['district'] as district FROM demo_user_events WHERE event_type = 'purchase' -- 结果：2 条 -- - 用户 10003: "智能手机 X1" → 上海市浦东新区 -- - 用户 10001: "纯棉 T 恤" → 北京市朝阳区
💡 亮点

支持多层嵌套访问 data['key']['nested']，直接获取深层数据

测试 5：按用户聚合购买记录 ✅

-- 业务场景：推荐系统要"用户历史购买列表" SELECT user_id, TO_JSON(COLLECT_LIST( STRUCT( event_data['product_name'], event_data['product_price'] ) )) AS purchases_json FROM demo_user_events WHERE event_type = 'purchase' GROUP BY user_id -- 结果：2 个用户 -- - 用户 10001: [{"col1":"纯棉 T 恤","col2":99}] -- - 用户 10003: [{"col1":"智能手机 X1","col2":4999}]

测试 6：动态字段提取 ✅

-- 业务场景：商品列表页要显示"关键属性"（不同类目显示不同） SELECT product_id, product_name, category_id, CASE WHEN category_id = 10 THEN CAST(attributes['material'] AS STRING) WHEN category_id = 20 THEN COALESCE( CAST(attributes['ram_gb'] AS STRING), CAST(attributes['driver_mm'] AS STRING) ) WHEN category_id = 30 THEN CAST(attributes['author'] AS STRING) END as key_attribute FROM demo_products -- 结果：5 条 -- - 纯棉 T 恤：100% 纯棉 -- - 牛仔裤：98% 棉 2% 弹性纤维 -- - 智能手机 X1: 12 -- - 无线耳机 Pro: 40 -- - Python 编程入门：张三

⚠️ 第四章：踩坑指南（真金白银换来的）

坑 1：JSON 字段不能直接用于 GROUP BY
-- ❌ 错误：会报错 "type 'json' is not supported in GROUP BY" SELECT attributes['material'], COUNT(*) FROM products GROUP BY attributes['material']; -- ✅ 正确：先 CAST 转换 SELECT CAST(attributes['material'] AS STRING), COUNT(*) FROM products GROUP BY CAST(attributes['material'] AS STRING);
坑 2：数组函数不能直接用于 JSON 数组
-- ❌ 错误：会报错 "invalid type 'json' of argument 1" SELECT * FROM products WHERE ARRAY_CONTAINS(attributes['colors'], '白色'); -- ✅ 正确：转成 STRING 后用 LIKE SELECT * FROM products WHERE CAST(attributes['colors'] AS STRING) LIKE '%白色%';
坑 3：不同类型不能直接 COALESCE
-- ❌ 错误：会报错 "cannot find appropriate common type" SELECT COALESCE(attributes['ram_gb'], attributes['driver_mm']) -- ✅ 正确：都转成 STRING SELECT COALESCE( CAST(attributes['ram_gb'] AS STRING), CAST(attributes['driver_mm'] AS STRING) )
坑 4：写入时必须用 PARSE_JSON
-- ❌ 错误：直接写字符串会报错 INSERT INTO products VALUES (..., '{"key": "value"}'); -- ✅ 正确：用 PARSE_JSON 或 CAST INSERT INTO products VALUES (..., PARSE_JSON('{"key": "value"}')); INSERT INTO products VALUES (..., CAST('{"key": "value"}' AS JSON));
坑 5：访问语法是方括号，不是冒号
-- ❌ 错误：Snowflake 的冒号语法不支持 SELECT data:key FROM table; -- ✅ 正确：方括号语法 SELECT data['key'] FROM table;

🎯 第六章：最佳实践

1. 混合建模：核心字段结构化 + 扩展字段 JSON

CREATE TABLE products ( -- 核心字段（结构化，用于筛选、JOIN、索引） product_id BIGINT PRIMARY KEY, category_id INT, brand STRING, base_price DECIMAL(18,2), -- 扩展字段（JSON，用于灵活存储） attributes JSON );

2. 对频繁查询的 JSON 字段创建生成列

-- 经常按"内存大小"筛选手机？ ALTER TABLE products ADD COLUMN ram_gb INT GENERATED ALWAYS AS ( CAST(attributes['ram_gb'] AS INT) ); CREATE INDEX idx_ram ON products(ram_gb); -- 查询时直接用生成列 SELECT * FROM products WHERE ram_gb >= 12;

3. 文档化 JSON Schema

📝 attributes 字段说明

服装类目 (category_id=10)

  • material: STRING - 材质
  • colors: ARRAY<STRING> - 颜色列表
  • sizes: ARRAY<STRING> - 尺码列表

电子类目 (category_id=20)

  • ram_gb: INT - 内存 (GB)
  • storage_gb: INT - 存储 (GB)
  • screen_size_inch: DOUBLE - 屏幕尺寸 (英寸)

4. 什么时候该用 JSON？

场景 推荐方案 原因
电商商品属性 ✅ JSON 不同类目属性差异大，频繁新增
用户行为日志 ✅ JSON 事件类型多，每种事件字段不同
IoT 设备数据 ✅ JSON 设备类型多样，上报格式不统一
核心业务字段 ❌ 结构化 如订单金额、用户 ID，需要强约束
频繁 JOIN 的字段 ❌ 结构化 JSON 字段 JOIN 性能不如结构化列

🎯 总结

用 JSON 字段代替大宽表，本质上是：

用查询时的少量类型转换，换取开发时的大量时间
用 JSON 的灵活性，换取 Schema 的稳定性
用列式存储的高效，换取稀疏数据的存储浪费
💬 最后一句真心话

如果你的业务字段经常变，或者不同实体的属性差异大，别犹豫，用 JSON。

如果字段固定、查询频繁、需要强约束，老老实实建列。
