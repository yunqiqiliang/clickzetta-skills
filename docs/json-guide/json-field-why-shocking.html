3. 本地化优化：国内访问延迟<50ms，成本是 Snowflake 的 1/3-1/5

数美科技 2PB 数据验证：存储↓68%，计算↓66%，查询从数十分钟→秒级。

大宽表的真实问题

某电商商品表，203 列：

CREATE TABLE products_old ( product_id BIGINT, -- 服装类目属性 (50 列) shirt_size STRING, shirt_color STRING, shirt_material STRING, -- ... 还有 47 列 -- 电子产品属性 (50 列) phone_ram INT, phone_storage INT, phone_screen DOUBLE, -- ... 还有 47 列 -- 图书类目属性 (50 列) book_author STRING, book_isbn STRING, book_pages INT, -- ... 还有 47 列 -- 总计：203 列 );
203
总列数
90%
NULL 值占比
3 天
新增字段等待
核心矛盾：
  • 业务需要灵活性（不同类目属性不同），但关系型数据库需要固定 Schema
  • 每次新增字段都要 DDL 变更 → DBA 审批 → 变更窗口 → 平均 3 天
  • 90% 的列在 90% 的行里是 NULL，存储浪费，查询还要扫描这些 NULL 列的元数据

JSON 方案：表结构

CREATE TABLE demo_products ( product_id BIGINT, category_id INT, product_name STRING, base_price DECIMAL(18,2), stock_quantity INT, brand STRING, origin_country STRING, attributes JSON, -- 所有扩展属性 created_at TIMESTAMP, updated_at TIMESTAMP, version INT ); -- 总共：11 列（原来是 203 列）
核心思路：核心字段结构化（用于筛选、JOIN、索引）+ 扩展字段 JSON 化（用于灵活存储）

数据写入示例

-- 服装：attributes 里存材质、颜色、尺码 INSERT INTO demo_products VALUES ( 1001, 10, '纯棉 T 恤', 99.00, 500, '优衣库', '中国', PARSE_JSON('{ "material": "100% 纯棉", "colors": ["白色", "黑色", "灰色"], "sizes": ["S", "M", "L", "XL"], "style": "休闲", "season": ["春", "夏", "秋"] }'), current_timestamp(), current_timestamp(), 1 ); -- 手机：attributes 里存 CPU、内存、屏幕 INSERT INTO demo_products VALUES ( 2001, 20, '智能手机 X1', 4999.00, 100, '小米', '中国', PARSE_JSON('{ "cpu": "骁龙 8 Gen 2", "ram_gb": 12, "storage_gb": 256, "screen_size_inch": 6.7, "battery_mah": 4500 }'), current_timestamp(), current_timestamp(), 1 );

查询测试：6 个真实场景（已验证）

测试 1：查询白色服装

-- 业务场景：运营要搞"白色系"促销活动
SELECT product_id, product_name, base_price
FROM demo_products
WHERE category_id = 10
AND CAST(attributes['colors'] AS STRING) LIKE '%白色%';
-- 结果：1 条 → 纯棉 T 恤 (¥99.00)
关键点

JSON 字段访问后需要 CAST(... AS STRING) 才能进行字符串比较。

测试 2：查询大内存手机

SELECT product_id, product_name, CAST(attributes['ram_gb'] AS INT) as ram_gb
FROM demo_products
WHERE category_id = 20
AND CAST(attributes['ram_gb'] AS INT) > 8;
-- 结果：1 条 → 智能手机 X1 (12GB RAM)

测试 3：统计各材质服装数量

SELECT CAST(attributes['material'] AS STRING), COUNT(*)
FROM demo_products
WHERE category_id = 10
GROUP BY CAST(attributes['material'] AS STRING);
-- 结果：2 种材质
⚠️ 踩坑：JSON 字段不能直接用于 GROUP BY

报错信息：type 'json' is not supported in GROUP BY（已验证）

测试 4：查询购买事件的收货地址（嵌套 JSON）

SELECT event_id, user_id,
event_data['product_name'],
event_data['shipping_address']['city'],
event_data['shipping_address']['district']
FROM demo_user_events
WHERE event_type = 'purchase';
-- 结果：2 条 → 上海市浦东新区、北京市朝阳区

测试 5：按用户聚合购买记录（JSON 聚合）

SELECT user_id,
TO_JSON(COLLECT_LIST(
STRUCT(event_data['product_name'], event_data['product_price'])
)) AS purchases_json
FROM demo_user_events
WHERE event_type = 'purchase'
GROUP BY user_id;
-- 结果：2 个用户 → [{"col1":"纯棉 T 恤","col2":99}]

测试 6：动态字段提取

SELECT product_name, category_id,
CASE
WHEN category_id = 10 THEN CAST(attributes['material'] AS STRING)
WHEN category_id = 20 THEN COALESCE(
CAST(attributes['ram_gb'] AS STRING),
CAST(attributes['driver_mm'] AS STRING)
)
END as key_attribute
FROM demo_products;
-- 结果：5 条 → 100% 纯棉、12、40、张三

踩坑指南（已验证报错）

坑 1：JSON 字段不能直接用于 GROUP BY
-- ❌ 错误 SELECT attributes['material'], COUNT(*) FROM products GROUP BY attributes['material']; -- ✅ 正确 SELECT CAST(attributes['material'] AS STRING), COUNT(*) FROM products GROUP BY CAST(attributes['material'] AS STRING);

报错验证：type 'json' is not supported in GROUP BY

坑 2：ARRAY_CONTAINS 不能直接用于 JSON 数组
-- ❌ 错误 SELECT * FROM products WHERE ARRAY_CONTAINS(attributes['colors'], '白色'); -- ✅ 正确 SELECT * FROM products WHERE CAST(attributes['colors'] AS STRING) LIKE '%白色%';

报错验证：function 'ARRAY_CONTAINS' cannot be resolved, invalid type 'json'

坑 3：COALESCE 需要类型兼容
-- ✅ 可以：两个相同类型的 JSON 字段 SELECT COALESCE(attributes['ram_gb'], attributes['driver_mm']); -- ❌ 错误：JSON 和 STRING 类型不兼容 SELECT COALESCE(attributes['ram_gb'], attributes['material']); -- ✅ 正确：都转成 STRING SELECT COALESCE( CAST(attributes['ram_gb'] AS STRING), CAST(attributes['driver_mm'] AS STRING) )

验证结果：相同类型的 JSON 字段可以直接 COALESCE，不同类型需要先 CAST

坑 4：写入时必须用 PARSE_JSON
-- ❌ 错误 INSERT INTO products VALUES (..., '{"key": "value"}'); -- ✅ 正确 INSERT INTO products VALUES (..., PARSE_JSON('{"key": "value"}'));

报错验证：cannot write incompatible data to table... string to json

为什么云器让人震撼？三个技术领先点

1. 生成列 + 自动索引（技术代差）

什么时候该用 JSON？

场景 推荐方案 原因
电商商品属性（不同类目属性差异大） ✅ JSON 新增属性无需 DDL，直接写 JSON
用户行为日志（事件类型多，字段不同） ✅ JSON 每种事件字段不同，JSON 灵活存储
IoT 设备数据（设备类型多样） ✅ JSON 上报格式不统一，JSON 容忍结构变化
核心业务字段（订单金额、用户 ID） ❌ 结构化 需要强约束、频繁 JOIN、索引加速
最佳实践
  • 混合建模：核心字段结构化 + 扩展字段 JSON
  • 生成列索引：对频繁查询的 JSON 字段创建生成列，自动索引
  • 文档化 Schema：记录每个类目的 JSON 字段结构，避免"字段猜谜"

🎯 总结

用 JSON 字段代替大宽表，本质上是：

  • 用查询时的少量类型转换，换取开发时的大量时间
  • 用 JSON 的灵活性，换取 Schema 的稳定性
  • 用列式存储的高效，换取稀疏数据的存储浪费

云器 Lakehouse 的 JSON 能力特点：

  • ✓ 嵌套访问 data['key']['nested']
  • ✓ JSON 聚合 TO_JSON(COLLECT_LIST(...))
  • ✓ 数组操作（需 CAST 后用 LIKE）
  • ✓ 生成列 + 自动索引（核心技术优势）
  • ✓ 存算分离（500TB 弹性扩展）
  • ⚠ GROUP BY 需要先 CAST
  • ⚠ 写入需要用 PARSE_JSON()

所以，云器 JSON 能力的定位：

在中国生产环境这个特定场景下，综合性能最优的 JSON 解决方案。

功能不是最强，但在生成列索引、存算分离、国内访问性能、成本这四个关键维度上，
云器提供了最匹配中国团队需求的整体解决方案。
