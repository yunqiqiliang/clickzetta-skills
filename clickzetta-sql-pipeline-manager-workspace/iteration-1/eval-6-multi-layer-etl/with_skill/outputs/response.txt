三层 ETL 动态表设计方案：ODS → DWD → DWS

## 架构设计思路

根据决策树，多层 ETL（ODS → DWD → DWS）的标准方案是：
- DWD 层：设置 TARGET_LAG = '1 minutes'，作为整条链路的驱动节点
- DWS 层：设置 TARGET_LAG = DOWNSTREAM，跟随 DWD 刷新完成后自动触发

这样 DWS 不会独立轮询，而是在 DWD 每次刷新完成后立即被驱动，端到端延迟约为 1 分钟 + DWS 自身计算耗时。

---

## 完整 SQL

### 前提：ODS 原始层（普通表，由上游写入）

ODS 层通常是普通表，由 Pipe（Kafka/对象存储）或业务系统直接写入，不需要是动态表。
以订单表为例：

```sql
CREATE TABLE IF NOT EXISTS ods.orders (
  order_id    STRING,
  user_id     STRING,
  amount      DECIMAL(10, 2),
  status      STRING,
  region      STRING,
  created_at  TIMESTAMP
);
```

---

### DWD 清洗层（每分钟刷新）

```sql
CREATE OR REPLACE DYNAMIC TABLE dwd.orders_clean
  TARGET_LAG = '1 minutes'
  VCLUSTER = default_ap
  COMMENT = 'DWD 清洗层：过滤无效数据，标准化字段'
AS
SELECT
  order_id,
  user_id,
  amount,
  UPPER(TRIM(status))      AS status,
  LOWER(TRIM(region))      AS region,
  created_at,
  DATE(created_at)         AS dt,
  DATE_TRUNC('hour', created_at) AS hour
FROM ods.orders
WHERE amount > 0
  AND order_id IS NOT NULL
  AND user_id  IS NOT NULL;
```

---

### DWS 聚合层（跟随 DWD 刷新）

```sql
CREATE OR REPLACE DYNAMIC TABLE dws.order_stats_hourly
  TARGET_LAG = DOWNSTREAM
  VCLUSTER = default_ap
  COMMENT = 'DWS 聚合层：按小时、地区统计订单指标'
AS
SELECT
  dt,
  hour,
  region,
  status,
  COUNT(*)                        AS order_cnt,
  COUNT(DISTINCT user_id)         AS uv,
  SUM(amount)                     AS total_amount,
  AVG(amount)                     AS avg_amount
FROM dwd.orders_clean
GROUP BY dt, hour, region, status;
```

---

## 刷新链路说明

```
ods.orders（普通表，持续写入）
    ↓
dwd.orders_clean   TARGET_LAG = '1 minutes'   ← 每分钟检测 ODS 变化，增量刷新
    ↓
dws.order_stats_hourly   TARGET_LAG = DOWNSTREAM   ← DWD 刷新完成后立即触发
```

- DOWNSTREAM 的含义：DWS 不设独立调度周期，完全由上游 DWD 驱动。每当 DWD 完成一次刷新，DWS 自动跟进一次。
- 端到端延迟：约等于 DWD 的 TARGET_LAG（1 分钟）加上 DWS 聚合计算耗时。
- 如果 DWS 下游还有更多层，同样设置 TARGET_LAG = DOWNSTREAM 即可继续级联。

## 验证命令

```sql
-- 查看动态表是否创建成功
SHOW TABLES IN dwd WHERE is_dynamic = true;
SHOW TABLES IN dws WHERE is_dynamic = true;

-- 查看 DWD 刷新历史（确认每分钟在刷新）
SHOW DYNAMIC TABLE REFRESH HISTORY dwd.orders_clean LIMIT 10;

-- 查看 DWS 刷新历史（确认跟随 DWD）
SHOW DYNAMIC TABLE REFRESH HISTORY dws.order_stats_hourly LIMIT 10;

-- 查看表结构和调度配置
DESC DYNAMIC TABLE dwd.orders_clean;
DESC DYNAMIC TABLE dws.order_stats_hourly;

-- 验证数据
SELECT * FROM dws.order_stats_hourly ORDER BY dt DESC, hour DESC LIMIT 20;
```

## 运维操作

```sql
-- 暂停整条链路（先暂停下游，再暂停上游）
ALTER DYNAMIC TABLE dws.order_stats_hourly SUSPEND;
ALTER DYNAMIC TABLE dwd.orders_clean SUSPEND;

-- 恢复（先恢复上游，再恢复下游）
ALTER DYNAMIC TABLE dwd.orders_clean RESUME;
ALTER DYNAMIC TABLE dws.order_stats_hourly RESUME;

-- 调整 DWD 刷新频率（如改为 30 秒）
ALTER DYNAMIC TABLE dwd.orders_clean SET TARGET_LAG = '30 seconds';
```

---

## 关键注意事项

1. TARGET_LAG = DOWNSTREAM 只能用于有上游动态表的场景，DWS 的上游必须是动态表（不能是普通表）。
2. 修改 SQL 逻辑必须用 CREATE OR REPLACE DYNAMIC TABLE，ALTER 不支持修改 AS 子句。
3. 刷新失败不影响查询可用性，失败时返回上次成功版本的数据，可通过 SHOW DYNAMIC TABLE REFRESH HISTORY 排查错误。
4. 若 ODS 层数据量大，建议在 DWD 的 AS 查询中加 PARTITION BY (dt) 以支持分区增量刷新，降低每次刷新的计算量。
