动态表（Dynamic Table）和物化视图（Materialized View）的区别，以及每天凌晨跑一次聚合报表的场景选择建议

---

## 核心区别

### 物化视图（Materialized View）

- 本质是一个预计算并持久化存储的查询结果集。
- 刷新方式通常分为：全量刷新（COMPLETE）和增量刷新（INCREMENTAL/FAST）。
- 刷新触发方式：手动触发、定时调度（CRON）、或基于源表变更触发（部分数据库支持）。
- 定义时绑定一条 SQL 查询，刷新时重新执行该查询并覆盖或合并结果。
- 适合查询加速、报表预聚合等场景，查询时直接读取已物化的数据，性能高。
- 对 SQL 语法支持较为严格，增量刷新对查询结构有限制（如不能含某些聚合或 JOIN 类型）。

### 动态表（Dynamic Table）

- 是一种声明式的数据管道抽象，定义"目标状态"而非"刷新过程"。
- 系统自动追踪上游数据变化，按照设定的 lag（延迟目标）持续或周期性地将结果更新到目标表。
- 刷新策略由系统根据 lag 目标自动决定（增量或全量），用户无需手动管理调度。
- 更适合构建流式或近实时的数据管道，支持链式依赖（多个动态表串联）。
- 对复杂 SQL（多层聚合、窗口函数、复杂 JOIN）的支持通常比物化视图更灵活。
- 底层会维护变更追踪元数据，存储和计算开销相对更高。

---

## 关键对比维度

| 维度           | 物化视图                        | 动态表                          |
|--------------|-------------------------------|-------------------------------|
| 刷新控制       | 手动 / 定时 CRON / 触发器        | 系统自动，基于 lag 目标            |
| 增量能力       | 有限制，依赖 SQL 结构             | 系统自动判断，更灵活               |
| 适用场景       | 查询加速、固定报表预聚合            | 数据管道、近实时/批量数据流转        |
| 调度灵活性     | 高（可精确控制执行时间）            | 较低（由系统决定，lag 是目标而非精确时间）|
| SQL 复杂度支持 | 受限（增量刷新有约束）             | 更宽松                          |
| 运维复杂度     | 低，行为可预期                   | 较低，但调试链路更复杂              |
| 存储开销       | 仅存结果                        | 结果 + 变更追踪元数据              |

---

## 你的场景：每天凌晨跑一次聚合报表

推荐使用：物化视图（Materialized View）

理由：

1. 执行时间明确：你需要在凌晨某个固定时间点触发，物化视图支持精确的 CRON 调度，行为可预期。动态表的 lag 机制是"尽量在 X 时间内刷新"，不保证精确的执行时间窗口。

2. 批量全量聚合：每天一次的聚合报表通常是全量重算或按日期分区增量，物化视图的全量刷新（COMPLETE）模式完全满足需求，逻辑简单清晰。

3. 资源可控：凌晨批跑场景下，你希望在低峰期集中消耗资源，物化视图的定时刷新可以精确控制资源使用窗口。动态表可能在非预期时间触发刷新，干扰资源规划。

4. 运维简单：物化视图的刷新历史、状态监控更直观，出问题容易排查。

5. 动态表更适合的场景是：需要近实时（分钟级）更新、有多层数据管道依赖、或者数据是流式持续写入的情况。你的场景是每日批量，不需要这些能力。

---

## 建议实现方式（物化视图）

```sql
-- 示例：每天凌晨 2 点全量刷新聚合报表
CREATE MATERIALIZED VIEW daily_sales_report
REFRESH COMPLETE
START WITH TRUNC(SYSDATE) + 2/24          -- 今天凌晨 2 点开始
NEXT TRUNC(SYSDATE + 1) + 2/24            -- 之后每天凌晨 2 点
AS
SELECT
    date_trunc('day', order_time) AS report_date,
    region,
    product_category,
    SUM(amount)   AS total_amount,
    COUNT(*)      AS order_count
FROM orders
GROUP BY 1, 2, 3;
```

具体语法依你使用的数据库（如 ClickZetta / Snowflake / Oracle 等）略有差异，但核心思路一致：定义查询 + 设置定时全量刷新。

---

## 总结

每天凌晨跑一次聚合报表 → 用物化视图，定时全量刷新，简单可控。
动态表留给需要近实时、流式、多级管道的场景。
