排查动态表 dws.order_hourly 刷新失败的步骤如下：

## 第一步：查看刷新历史，获取错误详情

```sql
SHOW DYNAMIC TABLE REFRESH HISTORY dws.order_hourly LIMIT 10;
```

这条命令会返回最近 10 次刷新记录，包括：
- 刷新状态（成功 / 失败）
- 错误信息（失败原因）
- 耗时
- 触发方式（自动 / 手动）
- 增量行数

重点看 `error_message` 或 `state` 字段，定位具体报错。

---

## 第二步：根据错误类型对症处理

### 情况 1：`VCluster not available`
计算集群未启动或名称错误。

```sql
-- 确认动态表当前配置的 VCLUSTER
DESC DYNAMIC TABLE dws.order_hourly;

-- 如需修改计算集群
ALTER DYNAMIC TABLE dws.order_hourly SET VCLUSTER = default_ap;
```

### 情况 2：SQL 查询报错（如源表结构变更、字段不存在）
源表 `dwd.orders_clean` 的字段可能被修改或删除。

```sql
-- 查看动态表的完整定义
SHOW CREATE TABLE dws.order_hourly;

-- 检查源表结构是否有变化
DESC TABLE dwd.orders_clean;
```

如果 SQL 逻辑需要修改，必须用 `CREATE OR REPLACE`（ALTER 不支持修改 AS 子句）：

```sql
CREATE OR REPLACE DYNAMIC TABLE dws.order_hourly
  TARGET_LAG = DOWNSTREAM
  VCLUSTER = default_ap
AS
SELECT
  DATE_TRUNC('hour', created_at) AS hour,
  status,
  COUNT(*) AS order_cnt,
  SUM(amount) AS total_amount
FROM dwd.orders_clean
GROUP BY 1, 2;
```

### 情况 3：上游动态表也失败（因为 TARGET_LAG = DOWNSTREAM）
`dws.order_hourly` 使用 `TARGET_LAG = DOWNSTREAM`，由上游驱动刷新。如果上游 `dwd.orders_clean` 刷新失败，下游也会失败。

```sql
-- 检查上游动态表的刷新历史
SHOW DYNAMIC TABLE REFRESH HISTORY dwd.orders_clean LIMIT 10;
```

先修复上游，下游会自动恢复。

### 情况 4：动态表被暂停
```sql
-- 查看表状态
SHOW TABLES IN dws WHERE is_dynamic = true;

-- 如果是 SUSPENDED 状态，恢复它
ALTER DYNAMIC TABLE dws.order_hourly RESUME;
```

---

## 第三步：确认是否支持增量刷新

```sql
EXPLAIN SELECT * FROM dws.order_hourly;
```

如果不支持增量刷新（退化为全量），在数据量大时可能导致超时失败。可以考虑：
- 简化 AS 查询（避免复杂的窗口函数或不支持增量的算子）
- 适当增大 TARGET_LAG，给刷新任务更多时间

---

## 快速排查总结

1. 先执行 `SHOW DYNAMIC TABLE REFRESH HISTORY dws.order_hourly LIMIT 10;` 看错误信息
2. 如果是集群问题 → 检查 VCLUSTER 配置
3. 如果是 SQL/源表问题 → 用 `CREATE OR REPLACE` 修复定义
4. 如果是上游失败 → 先排查 `dwd.orders_clean`
5. 如果是暂停状态 → `ALTER DYNAMIC TABLE dws.order_hourly RESUME;`

注意：刷新失败不影响表的可查询性，查询会返回上次成功刷新的数据版本。
