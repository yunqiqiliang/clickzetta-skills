排查动态表 dws.order_hourly 刷新失败的步骤

当动态表（Dynamic Table）刷新失败时，可以按以下步骤逐步排查：

---

1. 查看刷新任务的错误信息

首先查看动态表的刷新历史，获取具体的错误信息：

-- 查看动态表刷新历史（Clickzetta / Snowflake 语法）
SHOW DYNAMIC TABLES LIKE 'order_hourly' IN SCHEMA dws;

-- 查看刷新任务历史
SELECT *
FROM TABLE(INFORMATION_SCHEMA.DYNAMIC_TABLE_REFRESH_HISTORY(
    NAME => 'dws.order_hourly'
))
ORDER BY REFRESH_START_TIME DESC
LIMIT 20;

重点关注字段：
- STATE：刷新状态（FAILED / SUCCEEDED）
- STATE_MESSAGE：失败原因描述
- REFRESH_START_TIME / REFRESH_END_TIME：刷新时间段

---

2. 常见失败原因及排查方向

2.1 上游数据源问题
- 动态表依赖的基础表或上游动态表本身刷新失败
- 排查：检查上游表是否有数据、是否也存在刷新失败

SELECT *
FROM TABLE(INFORMATION_SCHEMA.DYNAMIC_TABLE_REFRESH_HISTORY(
    NAME => 'dws.upstream_table'
))
ORDER BY REFRESH_START_TIME DESC
LIMIT 10;

2.2 SQL 逻辑错误
- 动态表定义的 SQL 中存在除零、类型转换失败、NULL 处理异常等运行时错误
- 排查：手动执行动态表的定义 SQL，观察是否报错

-- 手动执行动态表的查询逻辑（去掉 CREATE DYNAMIC TABLE 部分）
SELECT ...
FROM dws.order_hourly_source
WHERE ...;

2.3 权限问题
- 刷新任务使用的角色/用户对上游表没有 SELECT 权限
- 排查：确认动态表 owner 或刷新使用的角色权限

SHOW GRANTS ON TABLE dws.order_source;

2.4 资源不足 / 超时
- 数据量过大导致刷新超时，或 warehouse 资源不足
- 排查：检查刷新耗时，考虑增大 warehouse 规格或优化 SQL

2.5 目标表结构变更
- 上游表字段被删除或类型变更，导致动态表 SQL 失效
- 排查：对比动态表定义 SQL 与上游表当前 schema

DESCRIBE TABLE dws.order_source;

2.6 滞后目标（Target Lag）配置问题
- Target Lag 设置过短，刷新频率过高导致积压
- 排查：检查当前 lag 配置是否合理

SHOW DYNAMIC TABLES LIKE 'order_hourly' IN SCHEMA dws;
-- 查看 TARGET_LAG 和 SCHEDULING_STATE 字段

---

3. 手动触发刷新并观察

在修复潜在问题后，可以手动触发一次刷新：

ALTER DYNAMIC TABLE dws.order_hourly REFRESH;

然后再次查询刷新历史确认是否成功。

---

4. 暂停与恢复

如果需要临时停止刷新（例如上游数据修复期间）：

-- 暂停
ALTER DYNAMIC TABLE dws.order_hourly SUSPEND;

-- 恢复
ALTER DYNAMIC TABLE dws.order_hourly RESUME;

---

5. 排查流程总结

1. 查刷新历史 → 获取 STATE_MESSAGE 错误信息
2. 根据错误信息定位：SQL 错误 / 权限 / 上游失败 / 资源超时
3. 手动执行定义 SQL 验证逻辑
4. 检查上游表状态和 schema
5. 修复后手动触发刷新验证

如果能提供 STATE_MESSAGE 的具体内容，可以更精准地定位问题。
