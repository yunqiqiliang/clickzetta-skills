# 三层 ETL 动态表设计方案：ODS → DWD → DWS

## 整体架构思路

使用动态表（Dynamic Table）构建三层 ETL 流水线，核心原则：
- ODS 层：原始数据落地，不做任何转换
- DWD 层：清洗、标准化，设置 1 分钟刷新周期
- DWS 层：聚合汇总，跟随 DWD 自动刷新（下游依赖上游）

---

## 第一层：ODS 原始层

ODS 层通常是外部表或普通表，作为数据源头，不使用动态表。
数据通过 Kafka、文件导入、CDC 等方式写入。

```sql
-- ODS 原始表（普通表，接收原始数据）
CREATE TABLE ods_orders_raw (
    order_id        STRING,
    user_id         STRING,
    product_id      STRING,
    amount          STRING,       -- 原始字段保留 STRING，清洗层再转换
    order_status    STRING,
    created_at      STRING,
    updated_at      STRING,
    _load_time      TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

---

## 第二层：DWD 清洗层（动态表，1 分钟刷新）

DWD 层对 ODS 数据做清洗、类型转换、去重、过滤无效数据。

```sql
-- DWD 清洗层动态表，每分钟刷新
CREATE DYNAMIC TABLE dwd_orders_clean
    TARGET_LAG = '1 minute'
    -- WAREHOUSE = 'your_warehouse'  -- 指定计算资源
AS
SELECT
    order_id,
    user_id,
    product_id,
    TRY_CAST(amount AS DECIMAL(18, 2))          AS amount,
    UPPER(TRIM(order_status))                    AS order_status,
    TRY_CAST(created_at AS TIMESTAMP)            AS created_at,
    TRY_CAST(updated_at AS TIMESTAMP)            AS updated_at,
    DATE(TRY_CAST(created_at AS TIMESTAMP))      AS order_date,
    CURRENT_TIMESTAMP                            AS dwd_load_time
FROM ods_orders_raw
WHERE
    order_id IS NOT NULL
    AND user_id IS NOT NULL
    AND TRY_CAST(amount AS DECIMAL(18, 2)) > 0   -- 过滤无效金额
    AND order_status NOT IN ('CANCELLED_TEST', 'DUMMY')  -- 过滤测试数据
QUALIFY ROW_NUMBER() OVER (
    PARTITION BY order_id
    ORDER BY TRY_CAST(updated_at AS TIMESTAMP) DESC
) = 1;  -- 去重，保留最新记录
```

关键参数说明：
- `TARGET_LAG = '1 minute'`：目标延迟 1 分钟，系统会尽量保证数据新鲜度在 1 分钟以内
- `QUALIFY ROW_NUMBER()`：去重逻辑，按 order_id 保留最新更新的记录

---

## 第三层：DWS 聚合层（动态表，跟随 DWD 刷新）

DWS 层基于 DWD 做聚合，使用 `DOWNSTREAM` 模式跟随上游自动刷新。

```sql
-- DWS 聚合层：按天、用户维度聚合订单
CREATE DYNAMIC TABLE dws_user_order_daily
    TARGET_LAG = DOWNSTREAM   -- 跟随上游 DWD 刷新，无需单独设置周期
    -- WAREHOUSE = 'your_warehouse'
AS
SELECT
    order_date,
    user_id,
    COUNT(DISTINCT order_id)            AS order_count,
    SUM(amount)                         AS total_amount,
    AVG(amount)                         AS avg_amount,
    MAX(amount)                         AS max_amount,
    MIN(amount)                         AS min_amount,
    COUNT(DISTINCT CASE WHEN order_status = 'COMPLETED' THEN order_id END)
                                        AS completed_order_count,
    SUM(CASE WHEN order_status = 'COMPLETED' THEN amount ELSE 0 END)
                                        AS completed_amount,
    CURRENT_TIMESTAMP                   AS dws_load_time
FROM dwd_orders_clean
GROUP BY
    order_date,
    user_id;
```

```sql
-- DWS 聚合层：按天、商品维度聚合
CREATE DYNAMIC TABLE dws_product_order_daily
    TARGET_LAG = DOWNSTREAM
AS
SELECT
    order_date,
    product_id,
    COUNT(DISTINCT order_id)            AS order_count,
    COUNT(DISTINCT user_id)             AS buyer_count,
    SUM(amount)                         AS total_amount,
    AVG(amount)                         AS avg_amount,
    CURRENT_TIMESTAMP                   AS dws_load_time
FROM dwd_orders_clean
GROUP BY
    order_date,
    product_id;
```

## 刷新机制说明

| 层级 | 表类型 | TARGET_LAG | 刷新触发方式 |
|------|--------|------------|-------------|
| ODS  | 普通表 | 无         | 外部写入     |
| DWD  | 动态表 | 1 minute   | 定时检测 ODS 变化 |
| DWS  | 动态表 | DOWNSTREAM | DWD 刷新后自动触发 |

`DOWNSTREAM` 的含义：DWS 不设独立刷新周期，而是在其依赖的 DWD 动态表完成刷新后，立即跟随刷新。这样整条链路的端到端延迟约为 1 分钟（DWD 刷新时间）+ DWS 计算时间。

---

## 完整建表顺序

```sql
-- Step 1: 建 ODS 原始表
CREATE TABLE ods_orders_raw ( ... );

-- Step 2: 建 DWD 动态表（依赖 ODS）
CREATE DYNAMIC TABLE dwd_orders_clean
    TARGET_LAG = '1 minute'
AS SELECT ... FROM ods_orders_raw;

-- Step 3: 建 DWS 动态表（依赖 DWD）
CREATE DYNAMIC TABLE dws_user_order_daily
    TARGET_LAG = DOWNSTREAM
AS SELECT ... FROM dwd_orders_clean;

CREATE DYNAMIC TABLE dws_product_order_daily
    TARGET_LAG = DOWNSTREAM
AS SELECT ... FROM dwd_orders_clean;
```

---

## 注意事项

1. 计算资源：每个动态表的刷新会消耗 warehouse 计算资源，建议 DWD 和 DWS 共用同一个 warehouse 以减少资源碎片。

2. 增量 vs 全量：动态表默认支持增量计算（如果查询可以增量化），对于含 `GROUP BY` 的聚合查询，系统会自动判断是否能增量处理，不能增量时退化为全量重算。

3. 监控刷新状态：
```sql
-- 查看动态表刷新状态
SHOW DYNAMIC TABLES LIKE 'dwd_%';
SHOW DYNAMIC TABLES LIKE 'dws_%';

-- 查看刷新历史
SELECT * FROM INFORMATION_SCHEMA.DYNAMIC_TABLE_REFRESH_HISTORY
WHERE name IN ('DWD_ORDERS_CLEAN', 'DWS_USER_ORDER_DAILY')
ORDER BY refresh_start_time DESC
LIMIT 20;
```

4. 手动触发刷新（调试用）：
```sql
ALTER DYNAMIC TABLE dwd_orders_clean REFRESH;
ALTER DYNAMIC TABLE dws_user_order_daily REFRESH;
```

5. 暂停/恢复：
```sql
ALTER DYNAMIC TABLE dwd_orders_clean SUSPEND;
ALTER DYNAMIC TABLE dwd_orders_clean RESUME;
```

---

## 设计总结

这套三层动态表架构的核心优势：
- 声明式定义，只需写 SELECT 逻辑，刷新调度由系统管理
- DWD 设 1 分钟 TARGET_LAG，保证数据新鲜度
- DWS 用 DOWNSTREAM，与 DWD 形成依赖链，避免数据不一致（DWS 永远基于最新的 DWD 数据聚合）
- 无需手动编排调度任务，降低运维复杂度
