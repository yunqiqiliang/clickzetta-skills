# Medallion Architecture on ClickZetta Lakehouse

以下是在 ClickZetta 上实现 Medallion Architecture 的完整 SQL 方案，数据源为 S3 Volume（s3_raw_vol），路径 /events/，JSON 格式。

---

## 1. Bronze 层 — 原始数据落地

Bronze 层直接读取 S3 上的原始 JSON 文件，以外部表或 COPY INTO 方式落地，保留原始字段，不做任何转换。

```sql
-- 创建 Bronze 层 Schema
CREATE SCHEMA IF NOT EXISTS bronze;

-- 创建 Bronze 原始事件表
CREATE TABLE IF NOT EXISTS bronze.events_raw (
    event_id    STRING,
    user_id     STRING,
    event_name  STRING,
    properties  STRING,   -- 原始 JSON 字符串，保留不解析
    created_at  STRING,   -- 保留原始字符串，不强制类型转换
    _loaded_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP()  -- 数据入库时间
);

-- 从 S3 Volume 加载原始 JSON 数据到 Bronze 层
COPY INTO bronze.events_raw (event_id, user_id, event_name, properties, created_at)
FROM VOLUME s3_raw_vol
LOCATION '/events/'
FILE_FORMAT = (
    TYPE = 'JSON'
    STRIP_OUTER_ARRAY = TRUE
);
```

> Bronze 层设计原则：
> - 保留所有原始字段，包括 properties 的原始 JSON 字符串
> - 新增 _loaded_at 记录入库时间，便于追溯
> - 不做任何业务逻辑处理，保证数据可回溯

---

## 2. Silver 层 — 数据清洗与展开

Silver 层基于 Bronze 层做清洗：过滤 event_name 为空的记录，并展开 properties 中的 page_url 和 referrer 字段。使用动态表（Dynamic Table）实现每 5 分钟自动刷新。

```sql
-- 创建 Silver 层 Schema
CREATE SCHEMA IF NOT EXISTS silver;

-- 创建 Silver 层动态表，每 5 分钟刷新
CREATE DYNAMIC TABLE silver.events_cleaned
    TARGET_LAG = '5 minutes'
    WAREHOUSE = '<your_warehouse_name>'
AS
SELECT
    event_id,
    user_id,
    event_name,
    -- 从 properties JSON 字符串中提取 page_url 字段
    GET_PATH(PARSE_JSON(properties), 'page_url')::STRING   AS page_url,
    -- 从 properties JSON 字符串中提取 referrer 字段
    GET_PATH(PARSE_JSON(properties), 'referrer')::STRING   AS referrer,
    -- 将 created_at 转换为标准 TIMESTAMP 类型
    TRY_TO_TIMESTAMP(created_at)                           AS created_at,
    _loaded_at
FROM bronze.events_raw
WHERE
    -- 过滤 event_name 为 NULL 或空字符串的记录
    event_name IS NOT NULL
    AND TRIM(event_name) != '';
```

> Silver 层设计说明：
> - TARGET_LAG = '5 minutes'：动态表每 5 分钟增量刷新一次
> - PARSE_JSON + GET_PATH：安全解析 properties JSON，提取嵌套字段
> - TRY_TO_TIMESTAMP：容错类型转换，避免格式异常导致整批失败
> - WHERE 过滤同时处理 NULL 和空字符串两种情况

---

## 3. Gold 层 — 聚合统计

Gold 层基于 Silver 层，按天统计每个 event_name 的触发次数和独立用户数。使用动态表跟随 Silver 层自动刷新。

```sql
-- 创建 Gold 层 Schema
CREATE SCHEMA IF NOT EXISTS gold;

-- 创建 Gold 层动态表，跟随 Silver 层刷新
CREATE DYNAMIC TABLE gold.event_daily_stats
    TARGET_LAG = DOWNSTREAM
    WAREHOUSE = '<your_warehouse_name>'
AS
SELECT
    DATE(created_at)            AS event_date,
    event_name,
    COUNT(*)                    AS event_count,
    COUNT(DISTINCT user_id)     AS unique_users
FROM silver.events_cleaned
WHERE
    created_at IS NOT NULL      -- 排除时间戳解析失败的记录
GROUP BY
    DATE(created_at),
    event_name;
```

> Gold 层设计说明：
> - TARGET_LAG = DOWNSTREAM：跟随上游 Silver 层的刷新节奏，Silver 刷新后 Gold 自动触发
> - COUNT(*) 统计事件总次数，COUNT(DISTINCT user_id) 统计独立用户数
> - WHERE created_at IS NOT NULL 排除 TRY_TO_TIMESTAMP 转换失败的脏数据

## 4. 数据流总览

```
S3 Volume (s3_raw_vol/events/*.json)
        |
        | COPY INTO (手动/调度触发)
        v
bronze.events_raw          -- 原始落地，无转换
        |
        | Dynamic Table, TARGET_LAG = 5 minutes
        v
silver.events_cleaned      -- 清洗 + 字段展开
        |
        | Dynamic Table, TARGET_LAG = DOWNSTREAM
        v
gold.event_daily_stats     -- 按天聚合统计
```

---

## 5. 查询示例

```sql
-- 查看最近 7 天各事件的每日统计
SELECT
    event_date,
    event_name,
    event_count,
    unique_users
FROM gold.event_daily_stats
WHERE event_date >= CURRENT_DATE() - INTERVAL '7 days'
ORDER BY event_date DESC, event_count DESC;

-- 查看 Silver 层某用户的清洗后数据
SELECT *
FROM silver.events_cleaned
WHERE user_id = '<target_user_id>'
ORDER BY created_at DESC
LIMIT 100;
```

## 6. 注意事项

1. Warehouse 名称：将 `<your_warehouse_name>` 替换为实际的 Virtual Warehouse 名称。
2. COPY INTO 调度：Bronze 层的 COPY INTO 需要通过任务调度（Task）或外部调度器定期触发，建议频率不低于 Silver 层的 5 分钟刷新间隔。
3. JSON 字段路径：如果 properties 中的字段路径有嵌套层级（如 `properties.meta.page_url`），需调整 GET_PATH 的路径参数，例如 `GET_PATH(PARSE_JSON(properties), 'meta.page_url')`。
4. 幂等性：COPY INTO 默认会跳过已加载的文件（基于文件名去重），保证 Bronze 层加载的幂等性。
5. 时区处理：如果 created_at 包含时区信息，建议在 Silver 层统一转换为 UTC，避免 Gold 层按天聚合时出现时区偏差。
