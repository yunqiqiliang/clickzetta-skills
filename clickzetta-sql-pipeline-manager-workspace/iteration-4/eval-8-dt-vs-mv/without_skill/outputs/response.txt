在 ClickZetta Lakehouse 中，动态表（Dynamic Table）和物化视图（Materialized View）都可以用于预计算和缓存查询结果，但它们的设计目标和刷新机制有所不同。

---

动态表（Dynamic Table）

动态表是 ClickZetta 中用于增量计算的对象。它定义了一个 SQL 查询，系统会根据配置的刷新策略自动维护结果集。

主要特点：
- 支持增量刷新（Incremental Refresh）：系统会追踪上游数据的变化，只处理新增或变更的数据，而不是每次全量重算。
- 支持完全刷新（Full Refresh）：也可以配置为每次全量重建。
- 刷新调度由系统管理，可以设置刷新频率（如每小时、每天等）。
- 适合持续更新的流式或批量数据管道场景。
- 可以链式依赖，构建多层数据管道。

物化视图（Materialized View）

物化视图是将视图的查询结果持久化存储的对象。

主要特点：
- 结果预先计算并存储，查询时直接读取，无需重新执行原始查询。
- 刷新方式通常为手动触发或基于规则触发，不像动态表那样有内置的自动调度机制。
- 在 ClickZetta 中，物化视图更多用于查询加速，即透明地替换原始查询以提升性能。
- 不一定支持增量刷新，通常是全量重建。

---

核心区别对比

| 维度         | 动态表                         | 物化视图                       |
|--------------|-------------------------------|-------------------------------|
| 刷新调度     | 内置自动调度，可配置频率        | 通常手动或外部触发              |
| 增量计算     | 支持增量刷新，效率更高          | 通常全量刷新                   |
| 使用场景     | 数据管道、ETL、持续聚合         | 查询加速、报表缓存              |
| 依赖链       | 支持多层动态表依赖              | 一般不支持链式依赖              |
| 控制粒度     | 开发者显式控制刷新逻辑          | 系统自动管理（透明替换）        |

---

你的场景建议：每天凌晨跑一次聚合报表

推荐使用动态表（Dynamic Table）。

理由：
1. 你有明确的调度需求（每天凌晨一次），动态表支持配置 TARGET_LAG 或定时刷新策略，可以精确控制刷新时间窗口。
2. 聚合报表通常涉及对大量数据的 GROUP BY、SUM、COUNT 等操作，动态表的增量刷新可以只处理当天新增数据，避免每次全量扫描，显著降低计算成本。
3. 动态表的结果是持久化存储的，报表查询直接读取预计算结果，响应速度快。
4. 如果报表有多个层次（如先聚合到小时级，再聚合到天级），可以用动态表链式构建，逻辑清晰。

示例思路：

CREATE DYNAMIC TABLE daily_sales_report
  TARGET_LAG = '1 day'
  WAREHOUSE = your_warehouse
AS
SELECT
  date_trunc('day', order_time) AS report_date,
  region,
  SUM(amount) AS total_amount,
  COUNT(*) AS order_count
FROM orders
GROUP BY 1, 2;

通过设置 TARGET_LAG = '1 day'，系统会保证数据延迟不超过一天，结合凌晨的刷新窗口，可以满足每日报表的需求。

如果你的聚合逻辑非常简单、数据量不大，且不需要增量能力，物化视图配合外部调度工具（如定时任务触发 REFRESH）也是可行的，但灵活性和效率不如动态表。
