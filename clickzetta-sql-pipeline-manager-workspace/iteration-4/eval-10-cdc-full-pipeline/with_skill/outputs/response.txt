## 分析

**数据来源**：已有表 ods.orders，有 INSERT 和 UPDATE（无 DELETE）
**层次需求**：ODS → DWD（最新状态去重）→ DWS（按天按 status 聚合）
**刷新频率**：近实时，延迟 < 1 分钟

**选择方案**：
- 源表有 UPDATE → 使用 `CREATE TABLE STREAM ... MODE = STANDARD` 捕获变更
- DWD 层：Dynamic Table 消费 Stream，过滤 `update_postimage` 和 `insert`，保留最新状态，`TARGET_LAG = '1 minutes'`
- DWS 层：Dynamic Table 基于 DWD 层聚合，`TARGET_LAG = DOWNSTREAM`（跟随上游自动触发）

---

## 完整端到端 SQL

### Step 1：创建 Schema

```sql
-- 确保 DWD 和 DWS schema 存在
CREATE SCHEMA IF NOT EXISTS dwd;
CREATE SCHEMA IF NOT EXISTS dws;
```

### Step 2：在 ODS 源表上创建 Table Stream（捕获 INSERT + UPDATE）

```sql
-- STANDARD 模式捕获所有变更（INSERT / UPDATE_PREIMAGE / UPDATE_POSTIMAGE）
CREATE TABLE STREAM ods.orders_stream
  ON TABLE ods.orders
  MODE = STANDARD
  COMMENT 'CDC stream on ods.orders，捕获 INSERT 和 UPDATE 变更';
```

### Step 3：DWD 层 —— 保留最新状态的订单

```sql
-- 消费 Stream，只取 insert 和 update_postimage（即最新值）
-- TARGET_LAG = '1 minutes' 保证近实时，延迟 < 1 分钟
CREATE OR REPLACE DYNAMIC TABLE dwd.orders_latest
  TARGET_LAG = '1 minutes'
  VCLUSTER = default_ap
  COMMENT 'DWD 层：从 Stream 消费最新订单状态（去除 update_preimage）'
AS
SELECT
  order_id,
  customer_id,
  amount,
  status,
  updated_at
FROM ods.orders_stream
WHERE _change_type IN ('insert', 'update_postimage');
```

### Step 4：DWS 层 —— 按天统计每个 status 的订单数和金额

```sql
-- TARGET_LAG = DOWNSTREAM，跟随 DWD 层自动触发，无需独立调度
CREATE OR REPLACE DYNAMIC TABLE dws.orders_daily_status_agg
  TARGET_LAG = DOWNSTREAM
  VCLUSTER = default_ap
  COMMENT 'DWS 层：按天、按 status 统计订单数和金额'
AS
SELECT
  DATE(updated_at)   AS stat_date,
  status,
  COUNT(*)           AS order_cnt,
  SUM(amount)        AS total_amount
FROM dwd.orders_latest
GROUP BY
  DATE(updated_at),
  status;
```

### Step 5：验证命令

```sql
-- 验证 Table Stream 已创建
SHOW TABLE STREAMS IN ods;

-- 查看 Stream 中待消费的变更数据（可选，用于调试）
SELECT COUNT(*) FROM ods.orders_stream;

-- 验证动态表已创建
SHOW TABLES IN dwd WHERE is_dynamic = true;
SHOW TABLES IN dws WHERE is_dynamic = true;

-- 查看 DWD 层刷新历史（确认近实时刷新正常）
SHOW DYNAMIC TABLE REFRESH HISTORY dwd.orders_latest LIMIT 5;

-- 查看 DWS 层刷新历史
SHOW DYNAMIC TABLE REFRESH HISTORY dws.orders_daily_status_agg LIMIT 5;

-- 抽查 DWD 层数据
SELECT * FROM dwd.orders_latest LIMIT 10;

-- 抽查 DWS 层聚合结果
SELECT * FROM dws.orders_daily_status_agg ORDER BY stat_date DESC, status LIMIT 20;
```

### Step 6：运维操作（按需使用）

```sql
-- 暂停（如集群维护）
ALTER DYNAMIC TABLE dwd.orders_latest SUSPEND;
ALTER DYNAMIC TABLE dws.orders_daily_status_agg SUSPEND;

-- 恢复
ALTER DYNAMIC TABLE dwd.orders_latest RESUME;
ALTER DYNAMIC TABLE dws.orders_daily_status_agg RESUME;

-- 调整 DWD 层延迟（如需更低延迟，改为 30 秒）
ALTER DYNAMIC TABLE dwd.orders_latest SET TARGET_LAG = '30 seconds';

-- 删除 Stream（如需重建）
DROP TABLE STREAM IF EXISTS ods.orders_stream;
```

## 架构说明

```
ods.orders（源表，持续 INSERT/UPDATE）
    │
    ▼
ods.orders_stream（Table Stream，MODE=STANDARD）
    │  捕获 insert / update_preimage / update_postimage
    ▼
dwd.orders_latest（Dynamic Table，TARGET_LAG='1 minutes'）
    │  过滤 _change_type IN ('insert','update_postimage')
    │  保留最新订单状态
    ▼
dws.orders_daily_status_agg（Dynamic Table，TARGET_LAG=DOWNSTREAM）
    │  按 DATE(updated_at) + status 分组
    │  COUNT(*) 订单数，SUM(amount) 金额
    ▼
  BI / 查询层
```

**延迟分析**：
- ods.orders 写入 → Stream 捕获：毫秒级
- Stream → DWD 层刷新：最大 1 分钟（TARGET_LAG = '1 minutes'）
- DWD → DWS 层刷新：DOWNSTREAM，DWD 刷新完立即触发，额外延迟极小
- 端到端总延迟：< 1 分钟，满足需求

**关键设计决策**：
1. 使用 `MODE = STANDARD` 而非 `APPEND_ONLY`，因为源表有 UPDATE
2. DWD 层只保留 `update_postimage`，丢弃 `update_preimage`，确保每次变更只保留最新值
3. DWS 层使用 `DOWNSTREAM` 避免独立调度，减少计算资源浪费，同时保证与 DWD 层数据一致性
4. DWS 层基于 `updated_at` 字段的日期分组，反映订单最后更新时间所在天的统计
