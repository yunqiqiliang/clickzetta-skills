排查动态表 dws.order_hourly 刷新失败，建议按以下步骤进行：

1. 查看刷新历史，确认失败信息

SHOW DYNAMIC TABLE REFRESH HISTORY dws.order_hourly LIMIT 20;

重点关注：
- STATUS 列：是否为 FAILED
- ERROR_MESSAGE 列：具体报错信息
- REFRESH_MODE 列：是增量（INCREMENTAL）还是全量（FULL）

2. 查看动态表当前状态和配置

DESC DYNAMIC TABLE dws.order_hourly;
DESCRIBE DYNAMIC TABLE dws.order_hourly EXTENDED;

确认：
- 表是否处于 SUSPENDED 状态（如果是，需要 RESUME）
- TARGET_LAG 配置是否合理
- VCLUSTER 是否存在且正常运行

3. 检查计算集群状态

确认 dws.order_hourly 使用的 VCLUSTER 是否在线、是否有足够资源。如果集群被暂停或资源不足，刷新任务会失败。

4. 检查上游数据源

如果 dws.order_hourly 的 AS 查询依赖其他表（如 ods 层的源表或上游动态表），需要确认：
- 上游表是否存在、是否有权限访问
- 如果上游也是动态表，检查上游的刷新状态

5. 验证 SQL 逻辑是否支持增量刷新

EXPLAIN SELECT * FROM dws.order_hourly;

如果输出显示不支持增量刷新，系统会回退到全量刷新。全量刷新数据量大时容易超时或 OOM，需要优化 SQL 或增大集群规格。

6. 常见失败原因及处理

- 集群不存在或已暂停：ALTER DYNAMIC TABLE dws.order_hourly SET VCLUSTER = <可用集群名>;
- 表被暂停：ALTER DYNAMIC TABLE dws.order_hourly RESUME;
- SQL 逻辑有问题（如引用了被删除的列）：用 CREATE OR REPLACE DYNAMIC TABLE 重建
- 资源不足/超时：升级 VCLUSTER 规格，或优化查询逻辑
- 上游表权限问题：检查当前用户对上游表的 SELECT 权限

7. 手动触发一次刷新验证

修复问题后，可以手动触发刷新确认是否恢复正常：

ALTER DYNAMIC TABLE dws.order_hourly REFRESH;

注意：刷新失败不影响表的可查询性，查询会返回上次成功刷新的数据版本。
